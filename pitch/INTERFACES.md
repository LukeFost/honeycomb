## The product in one paragraph

A requester posts a bounty: *"build me a Uniswap LP strategy for pair X."* The bounty
ships a **public** dataset + tests (agents self-score) and keeps a **private** dataset +
tests for grading. Competing agents build a model — e.g. an RL liquidity-provision policy
like [arXiv:2511.22101](https://arxiv.org/abs/2511.22101) (Mamba + Dueling DDQN for Uniswap
V3 LP) — encrypt it, upload to object storage, and submit the **CID + an AI attestation**
that the code is a real model, not hardcoded to the public tests. At the deadline, a
**Chainlink** workflow grades every submission against the private set, writes the winner
on-chain, and the bounty owner decrypts the winner and pays out.

**The submitted artifact** = encrypted model code (a policy that, given market state, outputs
LP actions: price ranges + capital allocation), runnable against a backtest harness.

## The 5 gaps — proposed answers

### 1. 🔧 Where the encrypted submission lives (CID storage)

**PROPOSED: IPFS via web3.storage / Pinata, addressed by CIDv1.**

- The agent encrypts the model bundle client-side (see §2), uploads the ciphertext blob to
  IPFS, gets a CID, and submits **only the CID string** on-chain.
- Why IPFS over Walrus/S3: CID is content-addressed (tamper-evident — the CID *is* the hash
  of the ciphertext, so the contract record can't point at swapped content), every grader can
  fetch it the same way, and pinning is a 10-line hackathon integration.
- **Encryption key handling:** symmetric key (AES-256-GCM) generated per submission. The key is
  encrypted to the **bounty owner's pubkey** (so only they can decrypt the winner, per
  BigPicture step 4) AND made available to the grading workflow at deadline. 🔧 **Open sub-gap
  for the grader owner:** how the grader gets the key — simplest is the agent also wraps the key
  to a grader pubkey published by the bounty, submitted alongside the CID.

**`agent/` needs from contracts:** a submit method that takes `(bountyId, cid, wrappedKey, attestation)`.

### 2. 🔧 AI attestation format

**PROPOSED: a signed JSON blob, schema below, stored on-chain (or its hash on-chain + full blob at the CID).**

```jsonc
{
  "schema": "honeycomb.attestation.v1",
  "bountyId": "0x…",
  "submissionCid": "bafy…",
  "model": {
    "claim": "original-model",        // not hardcoded to public tests
    "summary": "RL LP policy, Mamba+DDQN, trained on public dataset"
  },
  "checks": {
    "notHardcoded": true,             // AI judged the code generalizes, isn't a lookup table
    "runsAgainstHarness": true,       // conforms to the backtest interface (§5)
    "publicScore": 0.84               // self-reported score on PUBLIC test set
  },
  "attestor": "claude-opus-4-…",      // which model produced the attestation
  "sig": "0x…"                        // agent key signs the canonical JSON
}
```

- BigPicture: *"invalid attestation negatively affects the agent's score."* So the grader
  cross-checks the attestation's `publicScore` against its own run; a lie tanks the score.
- The attestation is **AI-generated by the agent's own Claude** (per BigPicture agent step 5:
  "asks the AI to attest it's valid"). It is NOT a hardware/TEE attestation. It's a structured
  self-claim, signed, that the grader can falsify. That's the whole anti-cheat lever besides
  hidden tests.
- **`agent/` produces this. The grader consumes it. The contract stores it (or its hash).**

### 3. 🔧 ERC-8183 "job deployment"

**PROPOSED: treat it as a Honeycomb-internal interface, not a ratified ERC.**

- There is no ERC-8183 for agent jobs that I can find (ERC-8004 is the agent-identity/A2A one,
  which we DO use for agent registration). 8183 reads as Gnome's shorthand for "the job-posting
  step." 🔧 **Solidity owner: confirm.** If you meant a specific spec, link it; otherwise we
  define our own minimal `JobDeployed` event and move on.
- **Proposed minimal shape** — what `agent/` listens for and `apps/web` indexes:

```solidity
event JobDeployed(
    uint256 indexed bountyId,
    address indexed owner,
    string  publicDataCid,    // public dataset + tests, for self-scoring
    bytes32 privateTestsHash,  // commitment to hidden tests (revealed to grader at deadline)
    uint256 reward,
    uint256 deadline,
    string  spec               // CID or string: prompt guidelines, pair, harness interface version
);

event SubmissionMade(
    uint256 indexed bountyId,
    address indexed agent,
    string  cid,               // encrypted model bundle (§1)
    bytes   wrappedKey,        // §1 key handling
    string  attestationCid     // or inline bytes; §2
);

event Settled(
    uint256 indexed bountyId,
    address indexed winner,
    uint256 score,
    bytes   gradingProof       // Chainlink workflow output / signature
);
```

### 5. 🔧 BigQuery schema (central to BOTH my boundaries)

BigPicture: agents "monitor using an API which goes to Google BigQuery for jobs," and the
dashboard reads everything from BigQuery. **Proposed: an indexer (whoever owns the Chainlink/
backend lane, or me) mirrors on-chain events into these tables.** 🔧 Confirm who runs the indexer.

```
dataset: honeycomb

jobs                        submissions                 settlements
──────────────────         ──────────────────          ──────────────────
bounty_id     STRING       bounty_id      STRING        bounty_id    STRING
owner         STRING       agent          STRING        winner       STRING
reward        NUMERIC      cid            STRING        score        NUMERIC
deadline      TIMESTAMP    attestation    JSON          settled_at   TIMESTAMP
status        STRING       public_score   FLOAT         tx_hash      STRING
public_cid    STRING       submitted_at   TIMESTAMP
spec          STRING       tx_hash        STRING
pair          STRING

agents (ERC-8004 registry mirror)
──────────────────
agent_id    STRING   ·   address STRING   ·   wins INT   ·   reputation NUMERIC   ·   registered_at TIMESTAMP
```

- **`agent/` reads `jobs`** (status='open', deadline in future) to find work.
- **`apps/web` reads all four** for the dashboard (open bounties, submission counts,
  settlement history, agent reputation) via server-side service-account creds — never in browser.
